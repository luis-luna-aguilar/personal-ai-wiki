Triaging newsletters

Newsletters are high-volume and mostly noise. Ingesting a newsletter as if it were a single article would bloat the wiki with one page per issue. Instead, newsletters go through a triage flow — the LLM extracts discrete signals, you tick the ones worth pursuing, and only those get ingested.

Why triage?

A typical issue of an AI newsletter has 10–20 signals: a new model release, three benchmark updates, a funding round, two vendor blog posts, and a "10 ways AI will change X" listicle. You probably want three of those in your wiki and don't care about the rest. Triage lets you be selective without having to read-and-reject each item.

The flow

  1. Drop the newsletter into raw/newsletters/ (or give the LLM a URL with "triage this URL").
  2. Say "triage this" or "triage the latest newsletter."
  3. The LLM writes a triage file to proposals/triage/YYYY-MM-DD-newsletter-name.md with a bulleted list of signals — each with a checkbox, an impact note, and a recommended action.
  4. Open the triage file in Obsidian. Tick the boxes for items you want to pursue. Leave the rest unchecked.
  5. Say "process the triage."
  6. For each checked item, the LLM generates a full ingest proposal (or applies a one-line note-only update). Unchecked items are ignored.

What a triage file looks like

---
type: triage
source: raw/newsletters/2026-04-09-the-batch.md
status: pending
---

# Triage: The Batch — 2026-04-09

## Signals

- [ ] **[coding]** Cursor 3 released, 74% SWE-bench (new top score)
    *Impact:* updates `state-of/coding.md`, `tools/cursor.md`; demotes previous leader line
    *Recommended:* full ingest

- [ ] **[models]** Anthropic announces Opus 4.7 preview
    *Impact:* new entry on `models/claude-opus.md`; bump `state-of/models.md`
    *Recommended:* full ingest

- [ ] **[marketing]** Vendor blog: "10 ways AI is changing X"
    *Impact:* nothing new
    *Recommended:* skip

- [ ] **[misc]** ChatGPT adds voice feature
    *Impact:* consumer side, outside current scope
    *Recommended:* note-only (one-liner in `trends/consumer-ai.md`)

Recommendation kinds

  • full ingest — generate a normal proposal for review. Use this when the signal genuinely updates or adds wiki content.
  • note-only — apply a one-line update directly without spawning a full proposal. Useful for confirmations of known trends or minor mentions.
  • skip — do nothing. Leave the box unchecked, or explicitly annotate.

Overriding recommendations

You can override the LLM's recommendation by editing the line. For example, downgrade a "full ingest" to "note-only" by changing the text:

- [x] **[coding]** Cursor 3 released, 74% SWE-bench
    *Recommended:* note-only  <-- edited from "full ingest"

When processing, the LLM respects what's in the file, not its original recommendation.

Processing

After you've marked up the triage file, say:

process the triage

or:

process proposals/triage/2026-04-09-the-batch.md

For each checked signal:

  • full ingest → a new proposal file appears in proposals/ for your review.
  • note-only → applied immediately to the target page. No proposal file.
  • skip / unchecked → ignored.

The triage file is then moved to proposals/triage/applied/. A log entry is written: ## [YYYY-MM-DD] triage | The Batch 2026-04-09 | 3 full, 1 note, 1 skipped.

Tips

  • Triage liberally, ingest selectively. A good triage takes two minutes — scan the signals, tick the 2–4 that matter, move on.
  • Note-only is underrated. For confirmations of known trends ("another company adopted X"), a one-line update to the trend page is usually enough.
  • Don't re-triage the same newsletter twice. If you need to add more signals after processing, drop the original file in again and ask for a new triage — or just do a manual one-off ingest.
  • Full ingests from triage follow the normal proposal flow. You still review each one before it touches the wiki.
