Fetching from URLs

Give the LLM a URL and it'll pull the content using a real Chromium browser via scripts/fetch_url.py. This bypasses most bot blocks and supports logged-in sites via a persistent profile.

Why not plain HTTP fetch?

Most news sites, Twitter, Substack, and paywalled content block simple HTTP fetches. A real browser with JavaScript execution, a realistic user agent, and cookie persistence gets through nearly all of it. This is why fetch_url.py uses Playwright-driven Chromium.

One-time setup

pip install playwright markdownify readability-lxml
playwright install chromium

Chromium lives wherever Playwright installs it (usually ~/Library/Caches/ms-playwright on macOS). The persistent browser profile lives at ~/.cache/ai-wiki-playwright by default — change via config.yml → fetch_url.profile_dir.

Usage (you telling the LLM)

Any of these work:

  • "fetch and ingest https://cursor.com/blog/cursor-3"
  • "ingest this URL: https://arxiv.org/abs/2404.12345"
  • "pull https://github.com/anthropics/claude-code and ingest as a repo"

The LLM decides the --type based on the URL shape (github → repo, arxiv → paper, twitter → tweet, otherwise → article), runs the script, then proceeds with a normal ingest proposal based on the saved raw file.

Usage (you running it directly)

You can also run it yourself before asking the LLM to ingest:

python scripts/fetch_url.py --type article --url https://cursor.com/blog/cursor-3
python scripts/fetch_url.py --type paper   --url https://arxiv.org/abs/2404.12345
python scripts/fetch_url.py --type repo    --url https://github.com/anthropics/claude-code
python scripts/fetch_url.py --type tweet   --url https://twitter.com/karpathy/status/1234

The script prints the saved file's relative path as its last line, so you can chain it.

Logged-in sites (Twitter, paywalls)

The script uses a persistent Chromium profile, so once you log in, cookies are saved and future runs work without re-authenticating.

To log in:

  1. Run the script once on any URL from the site you need to log into.
  2. If it's headless, run with the profile in headed mode — the simplest trick is to open the profile dir manually in Chromium: chromium --user-data-dir=~/.cache/ai-wiki-playwright (or let the LLM run a one-off interactive session).
  3. Log in as you normally would. Close the browser.
  4. Future runs of fetch_url.py will reuse the cookies.

Alternatively, run without the persistent profile via --no-profile for one-off fetches.

When the fetch fails

Some sites will still fail — very aggressive anti-bot (Cloudflare challenges), heavy JS frameworks with slow hydration, or sites that gate on WebGL fingerprints. Fallbacks:

  • Obsidian Web Clipper — clip the article manually in your real browser, drop into raw/articles/.
  • Paste the text — copy the content, paste it into a new raw/ file, then ingest.
  • Archive.org / archive.today — sometimes easier to fetch.
Do not fall back to Claude Code's built-in WebFetch tool. That's the thing fetch_url.py replaces — WebFetch gets blocked too often and can't execute JavaScript. The LLM is instructed to never use it for source ingestion.

Images in fetched content

When config.yml → fetch_url.download_images is on (the default), images referenced in the article will be downloaded to raw/assets/ and the markdown links rewritten to local paths. This means the LLM can later open an image to look at it directly, which matters for benchmark charts and screenshots.

Command reference

python scripts/fetch_url.py \
  --type {article,tweet,paper,podcast,repo,newsletter,note} \
  --url  https://...                 # required
  [--slug my-custom-slug]             # override auto-generated filename slug
  [--no-profile]                      # don't use the persistent Chromium profile
  [--profile-dir /path/to/profile]    # override profile location
