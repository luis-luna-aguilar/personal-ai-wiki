Proposals workflow

Every source goes through a proposal before it touches the wiki. This is the safety rail — you see exactly what's about to change, check what you agree with, edit what needs tweaking, and apply only the approved parts.

When a proposal appears

After you ask the LLM to ingest a source (or process a triage item marked "full ingest"), you'll get a message telling you a proposal has been written to proposals/YYYY-MM-DD-slug.md. Open it in Obsidian.

What a proposal contains

  1. Summary — two sentences describing the source.
  2. Intended changes — a checkbox list of every action the LLM wants to take (update X, create Y, spill Z to history, etc.).
  3. Page drafts — the full proposed markdown for every new or updated page, in fenced code blocks.
  4. Schema / vocabulary additions — when the LLM wants to introduce a new tag, domain, or subcategory. Requires its own checkbox.
  5. Open questions — points where the LLM is uncertain and wants your input.

How to review

  1. Read the summary. Is this source worth keeping at all? If not, say "reject it."
  2. Scan the intended changes. Does each change make sense? Tick the ones you agree with. Leave the rest unchecked.
  3. Edit drafts if needed. The page drafts are full markdown — you can rewrite sentences, delete paragraphs, change links. What's in the file when you apply is what gets written.
  4. Answer open questions. Edit the file to leave comments next to the LLM's questions.
  5. Apply. Say "apply it" or "apply proposals/xxx.md."
The file you edit is the source of truth. The LLM re-reads the proposal file at apply time — it doesn't remember what it originally wrote. Whatever's in the file when you apply is what gets written to the wiki.

Feedback conventions

The LLM reads the proposal file at apply time and uses these conventions to decide what to do. You don't have to follow them rigidly — they exist so you don't have to think about format — but knowing them helps if a draft isn't being applied the way you expected.

Checkboxes

  • [x] means include — the LLM will perform that action when you say apply.
  • [ ] means skip — unchecked items are silently dropped at apply time.
  • This applies to every Intended changes item AND every Schema / vocabulary additions item.

Editing drafts in place

Every page draft sits inside a fenced markdown code block under Page drafts. To change what gets written, edit the block directly — rewrite sentences, delete paragraphs, fix wikilinks, change frontmatter, whatever. The LLM uses the draft as it appears in the file at apply time, not what it originally wrote.

If you delete a draft entirely but leave the corresponding intended-change item checked, the LLM will stop and ask rather than guess.

Answering open questions

Open questions live at the bottom of the proposal under Open questions. Add your answer as an indented bullet directly under the question. Any prefix works — →, Comment:, plain text — the LLM picks it up either way.

- **Should we create a stub for this model?** ...
    - Yes, create the stub.

Free-form notes

For anything that doesn't fit a specific question or draft, add a ## Comments section at the bottom of the proposal and write whatever you want. The LLM reads it at apply time and adjusts its behavior. Use this for cross-cutting feedback, format complaints, scope changes, or "by the way, also do X."

Diff display in updates

For Create actions the LLM shows the full draft (there's nothing to diff against). For Update actions it tries to show only the delta — the section being added or replaced — rather than re-printing the whole page. Index updates are shown as just the new lines plus the replaced page-count block. If a draft format makes editing harder than it should, say so in a ## Comments section and the LLM will adjust.

Example proposal

---
type: proposal
source: raw/articles/2026-04-09-cursor-3-release.md
status: pending
created: 2026-04-09
---

# Proposal: Cursor 3 release

## Summary
Two-sentence TL;DR of the source.

## Intended changes

- [ ] **Update** `wiki/state-of/coding.md` — bump Cursor line, update as_of
    > **Before:** `- [[Cursor]] — 68% SWE-bench (as of 2026-03-22)`
    > **After:**  `- [[Cursor]] — 74% SWE-bench (as of 2026-04-09)`

- [ ] **Update** `wiki/tools/cursor.md` — version bump, new recent-change entry
    > See draft below

- [ ] **Create** `wiki/benchmarks/swe-bench.md`
    > See draft below

- [ ] **Create** `wiki/sources/articles/cursor-3-release.md`

## Page drafts

### wiki/tools/cursor.md (updated)
```
---
title: Cursor
type: tool
domains: [coding]
...
---
# Cursor
...(full content)...
```

### wiki/benchmarks/swe-bench.md (new)
```
...(full content)...
```

## Schema / vocabulary additions

- [ ] Add subcategory `ide-pair-programming` to `wiki/_schema/subcategories.md`

## Open questions

- Should I merge Cursor 3 content into existing `tools/cursor.md` (assumed yes),
  or create a separate page for v3?

How to apply

Once the file is ready, say:

apply it

or:

apply proposals/2026-04-09-cursor-3-release.md

The LLM will:

  1. Re-read the proposal file as it is now.
  2. Execute only the checked items.
  3. Use the drafts as they appear in the file — including any edits you made.
  4. Update wiki/index.md with any new pages.
  5. Update wiki/_schema/ for any approved vocabulary additions.
  6. Append a log entry to wiki/log.md.
  7. Move the proposal file to proposals/applied/.
  8. Report a short summary back to you.

How to reject

If a proposal isn't worth pursuing at all, say:

reject it

The file is moved to proposals/rejected/, no changes are made to the wiki, and a log entry is written.

Partial rejection

There's no such thing — a proposal with unchecked items is a partial apply. Uncheck anything you don't want, apply the rest, and the unchecked items effectively get rejected.

If you want to keep the proposal open for later review instead of applying immediately, just leave it alone. Proposals stay in proposals/ until you apply or reject them.

Editing drafts directly

The drafts in a proposal are fair game — you can:

  • Rewrite entire sections
  • Add or remove bullet points
  • Fix factual errors
  • Add wikilinks the LLM missed
  • Change the tone or phrasing

When you apply, the LLM writes exactly what's in the file, not what it originally drafted. This is how the dry-run becomes conversational — you're literally co-editing a document before it gets committed.

The proposals/ directory

proposals/
  2026-04-09-cursor-3-release.md     # pending — you haven't applied yet
  applied/
    2026-04-08-karpathy-tweet.md     # applied yesterday
  rejected/
    2026-04-05-vendor-blog.md        # rejected (wasn't worth ingesting)
  triage/
    2026-04-09-the-batch.md          # pending triage
    applied/
      2026-04-02-import-ai.md        # already processed

You can look back at applied/ to see what's been ingested. The files there are the exact final form you approved.
