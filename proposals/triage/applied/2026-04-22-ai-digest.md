- [x] **[agents, coding]** Shopify AI phase transition — near-100% adoption, December inflection, critique loops > parallel agents

    **What it is:** Full Latent Space podcast with Shopify CTO Mikhail Parakhin. Key data: near-universal AI tool adoption (~100% daily active workers), with a December 2025 inflection point where small model-quality improvements compounded into a phase change. CLI-first tools (Claude Code, Codex, internal River agent) are growing faster than IDE tools (Cursor, Copilot). Core insight on agent quality: running fewer agents with multi-turn critique loops (one agent generates, another model critiques, first agent redoes) beats running many agents in parallel — even though critique loops are slower. Shopify built their own PR review tool because no external tool spends enough on expensive models during review. 30% month-on-month PR merge growth means the real bottleneck is now CI/CD, test failures, and deployment rollback — not code generation. Three internal systems: Tangle (content-addressed ML workflow engine, dev→prod in one click), Tangent (auto-research agent loop — auto-ML via LLMs, democratizing experimentation to PMs), SimGym (customer simulation on historical data, 0.7 correlation target with add-to-cart events). Also uses Liquid AI for 30ms low-latency search query understanding and batch distillation tasks where Qwen alternatives lose on efficiency.

    **Why it matters:** This is the richest real-world enterprise AI adoption data point in months. The "critique loops > parallel swarms" framing is directly applicable to the wiki's agents and training pages. The CI/CD-as-bottleneck insight is new and practically important. Updates training/company-wide-ai-enablement, concepts/harness, state-of/agents; SimGym adds to trends/proprietary-data-becomes-model-moat.

- [x] **[agents, coding]** Tokenmaxxing as vanity metric — Meta Claudeonomics 60T tokens/month, gaming, and the lines-of-code parallel

    **What it is:** Meta built an internal leaderboard called Claudeonomics ranking 85,000 employees by AI token usage. In 30 days, the company consumed 60 trillion tokens — triple every book ever published. Jensen Huang stated he'd be concerned if a $500K engineer wasn't spending $250K/year on tokens (with the caveat that he sells the chips). Immediate consequence: employees began leaving agents idle to game rankings. At OpenAI, one engineer processed 210 billion tokens in a single week (enough to fill Wikipedia 33 times). This phenomenon is dubbed "tokenmaxxing." The newsletter draws the exact parallel to the early 2000s lines-of-code productivity mistake — a measurable proxy gets optimized at the expense of actual output. Reasoning models compound the problem: they generate tokens for inner monologue before producing output, making token counts a function of model architecture as much as actual work done.

    **Why it matters:** Strong cultural signal that token spend is being treated as a proxy for engineer productivity — and is already being gamed. This updates training/company-wide-ai-enablement (what not to measure) and reinforces the Shopify CTO's point about needing better outcome metrics than raw throughput. The tokenmaxxing framing is also a useful counter-narrative to the "more tokens = more AI adoption" story companies use to justify AI spend.
